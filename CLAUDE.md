## Scripts

The repo is published via **GitHub Pages** (`.nojekyll` at root; Settings → Pages → deploy from `main` / root), so scripts have a short canonical remote-invocation URL: `https://ohjho.github.io/dotfiles/scripts/<name>.py` (e.g. `uv run https://ohjho.github.io/dotfiles/scripts/convert_media.py --help`). The longer `raw.githubusercontent.com/ohjho/dotfiles/main/scripts/<name>.py` URL also still works.

- **`scripts/convert_media.py`** — Audio/video format converter built on `ffmpeg-python` (requires `ffmpeg` on `PATH`). Uses `typer` for the CLI and `loguru` for logging, with PEP 723 inline deps so it runs self-contained: `uv run scripts/convert_media.py INPUT --to FORMAT`. Accepts a single file, a directory, or a glob; `--output` is treated as a directory when it has no extension and a target file otherwise. Flags: `--to/-t` (required), `--output/-o`, `--recursive/-r`, `--audio-bitrate`, `--video-codec`, `--audio-codec`, `--dry-run/-n`, `--overwrite/-f`, `--verbose/-v`. Exit codes: `0` ok, `1` if any file failed, `2` if no inputs matched. Pure helpers (`normalize_format`, `build_output_path`, `build_output_kwargs`) carry doctests; per-file ffmpeg work is isolated in `convert_one` so `convert` is testable via an injected `convert_fn`.
  - Tests: `uv run --with ffmpeg-python --with typer --with loguru --with pytest pytest scripts/ --doctest-modules` (covers unit tests + doctests, no real ffmpeg needed).
- **`scripts/probe_media.py`** — Shows key information about a single video/audio file using `ffmpeg.probe` (a wrapper around `ffprobe`; requires `ffmpeg` on `PATH`). A friendlier replacement for `ffprobe -show_format -show_streams ... | grep`. Same stack as `convert_media.py` (`ffmpeg-python`/`typer`/`loguru`, PEP 723 inline deps): `uv run scripts/probe_media.py INPUT`. Accepts a **local path or a remote URL** (`http(s)`, `ftp`, `rtmp`, `rtsp`, ...; the existence check is skipped for URLs). Prints a readable summary (container, duration, bitrate/size, one labeled block per stream with codec/resolution/fps or sample-rate/channels) by default; `--json/-j` emits the full raw ffprobe JSON; `--verbose/-v` for debug logging. Two tiered integrity checks: `--check/-c` is a fast, remote-cheap **structural** check derived from the metadata already fetched (flags missing streams, zero/missing duration, video streams without dimensions) — it cannot see frame-level corruption; `--deep-check` runs a full decode pass (`ffmpeg -v error -i INPUT -f null -`) that catches bitstream corruption but is slow and downloads remote inputs in full (it supersedes `--check` when both are given). Exit codes: `0` ok, `1` probe failed or a check found problems, `2` file not found. Pure helpers (`is_remote_input`, `find_structural_issues`, `parse_decode_errors`, `format_frame_rate`, `format_duration`, `extract_stream_info`, `build_summary`) carry doctests; the side-effecting `ffmpeg` calls are isolated in `probe_file`/`run_integrity_check` so the formatting/parsing stays testable.
  - Tests: same invocation as `convert_media.py` above (`pytest scripts/ --doctest-modules`).
- **`scripts/list_s3.py`** — Lists S3 objects filtered by file suffix and prints them as keys, HTTPS URLs, or `s3://` URIs — a paginated, region-aware replacement for the `aws s3api list-objects-v2 ... --query "Contents[?ends_with(Key,'.mp4')...]" | sed 's#^#https://...#'` one-liner. Built on `boto3` with `typer`/`loguru` and PEP 723 inline deps: `uv run scripts/list_s3.py LOCATION`. `LOCATION` is either an `s3://bucket/prefix` URI or a bare bucket name; credentials/region resolve through boto3's normal chain (env vars, `~/.aws/config`, profiles). Unlike the bash one-liner it **paginates past the first 1000 keys** (via `get_paginator("list_objects_v2")`) and auto-detects the bucket region for URLs. Suffix filtering is client-side (S3 has no server-side suffix filter), case-insensitive, and accepts repeated or comma-separated values (`-s .mp4 -s mov` or `-s mp4,mov`); omit it to list everything. Flags: `--prefix/-p` (overrides any prefix in LOCATION), `--suffix/-s`, `--format/-f` (`key` default, `url`, `uri`), `--limit/-n`, `--region` (else auto-detected, only for `url`), `--profile`, `--verbose/-v`. Output is newline-separated on stdout (pipe-friendly); the count goes to stderr. URLs use virtual-hosted-style, falling back to path-style for dotted bucket names (TLS wildcard-cert mismatch). Exit codes: `0` ok, `1` AWS/boto3 error, `2` no bucket in LOCATION. Pure helpers (`parse_location`, `normalize_suffix`, `normalize_suffixes`, `matches_suffix`, `normalize_region`, `build_object_url`, `render`) carry doctests; the boto3-free `iter_objects` generator takes an injected page source, with the only AWS calls isolated in `s3_pages`/`resolve_region` so listing/formatting stays testable.
  - Tests: `uv run --with boto3 --with typer --with loguru --with pytest pytest scripts/list_s3.py --doctest-modules` (doctests only, no live S3 needed).
- **`scripts/upload_imgbb.py`** — Uploads a single local image to [imgbb](https://api.imgbb.com) and prints its hosted URL — a one-command replacement for the manual `curl` dance (read the file, base64-encode it, `POST` it with the key). Built on `requests` with `typer`/`loguru` and PEP 723 inline deps: `uv run scripts/upload_imgbb.py IMAGE`. The image is base64-encoded and sent via `POST` to `https://api.imgbb.com/1/upload` (key as a query param, image as form data); on success the direct image URL (`data.url`) is printed to stdout. The API key is an **optional** flag (`--api-key/-k`) that defaults to the `IMGBB_API_KEY` env var. Flags: `--api-key/-k`, `--name` (stored filename; defaults to the image's stem), `--expiration` (auto-delete after N seconds, 60–15552000), `--json/-j` (emit imgbb's full raw response instead of just the URL), `--verbose/-v`. Output is the URL on stdout (pipe-friendly); the status line goes to stderr. Exit codes: `0` ok, `1` upload failed (network error, non-2xx, or `success: false`), `2` usage error (file not found, or no API key supplied/in env). Pure helpers (`to_base64`, `build_payload`, `extract_url`) carry doctests; the side-effecting `requests.post` is isolated in `upload_image` (which returns the full parsed JSON payload) and the file read in `read_image`, so the encoding/formatting stays testable.
  - Tests: `uv run --with requests --with typer --with loguru --with pytest pytest scripts/upload_imgbb.py --doctest-modules` (doctests only, no live imgbb/network needed).

# Guidelines
- update CLAUDE.md for each new features
- keep code modular to ensure ease in future refactoring
- use python and typer and loguru when writing scripts and make sure that it can be `uv run`
- use Google-style docstring for new functions and add a doctest compatible unit test if possible
